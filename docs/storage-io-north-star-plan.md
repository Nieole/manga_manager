# 外接机械盘 IO 优化北极星计划书

更新日期：2026-05-25

本文档针对“资源库位于外接机械移动硬盘时，启动扫描或重建封面导致 Windows 任务管理器磁盘活动时间长期 100%、但读取速度只有 5-10 MB/s，并且手动访问硬盘严重卡顿”的场景制定。优化目标不是单纯提高吞吐，而是降低随机 IO 压力、保护阅读交互延迟、让后台维护任务对系统可用性的影响可控。

## 问题定义

机械盘活动时间 100% 但吞吐不高，通常说明瓶颈在随机 IO、寻道、USB 桥接、硬盘队列和小块读写延迟，而不是顺序带宽。漫画管理场景中的扫描、打开归档、读取页数、解析 ComicInfo、计算 hash、提取封面、解码缩略图会制造大量小 IO 和跨归档随机访问。外接机械盘在这类负载下很容易被 IOPS 打满，导致资源管理器打开目录、移动文件、阅读翻页都变慢。

## 北极星目标

- 外接机械盘资源库进入低冲击模式后，后台扫描和封面重建不能明显阻塞阅读翻页。
- 日常增量扫描优先比较 path、mtime、size，未变化文件不打开归档。
- 封面生成、hash、元数据补全全部队列化，并按存储设备限制并发。
- 同一块外接机械盘同时只运行一个重 IO 任务。
- 阅读请求拥有最高优先级，扫描、封面、hash 能在阅读时降速或暂停。
- 页面磁盘缓存默认不落在漫画所在外接机械盘。
- 诊断页能解释“当前是谁在打满磁盘”，而不是只看到任务运行中。

## 设计原则

1. 保护交互优先于后台吞吐
   - 阅读、打开详情页、资源库基础浏览优先。
   - 扫描、封面重建、full hash 属于可延迟任务。

2. 按存储设备限流，而不是只按任务类型限流
   - 同一进程里不同任务可能访问同一块盘。
   - 必须按 library root 或 volume key 聚合调度。

3. 默认轻量，显式重建
   - 普通扫描只做低成本比较。
   - 只有用户选择 repair/rebuild 时才全量打开归档和重建派生数据。

4. 顺序化机械盘重 IO
   - 对外接 HDD，归档打开、封面生成、hash 默认并发为 1。
   - 对 SSD 可以允许更高并发。

5. 可观测再优化
   - 日志和性能面板必须记录打开归档数、hash 文件数、封面任务数、队列等待时间和阅读抢占次数。

## 阶段 1：存储介质策略与配置模型

目标：让系统知道某个资源库位于外接机械盘，并能自动套用低冲击策略。

### 改造项

1. 新增资源库存储策略
   - `storage_profile`: `auto`、`ssd`、`hdd_external`、`network`、`custom`。
   - 默认 `auto`，允许用户手动覆盖。
   - 外接机械盘推荐选择 `hdd_external`。

2. 新增 IO 策略配置
   - 扫描归档打开并发。
   - 封面生成并发。
   - hash 并发。
   - 是否阅读时暂停后台重 IO。
   - 是否仅空闲时运行封面/identity 任务。
   - 是否禁用同盘页面磁盘缓存。

3. 增加配置 UI
   - 资源库设置中显示“存储介质策略”。
   - 提供“外接机械盘低冲击模式”快捷选项。
   - 展示每种策略的主要行为，不展示冗长说明文本。

### 建议配置形态

```yaml
libraries:
  - path: "E:/Manga"
    storage_profile: hdd_external
    io_policy:
      scan_concurrency: 1
      archive_open_concurrency: 1
      cover_concurrency: 1
      hash_concurrency: 1
      pause_background_when_reading: true
      idle_only_heavy_tasks: true
      disable_same_disk_page_cache: true
```

### 验收标准

- 每个资源库能读取并保存存储策略。
- 外接机械盘策略默认将重 IO 并发限制为 1。
- 阅读时后台任务可被暂停或降速。
- 配置变更不需要重启即可生效。

## 阶段 2：统一存储 IO 调度器

目标：扫描、封面、hash、阅读不再各自无序访问同一块盘，而是进入统一调度。

### 改造项

1. 新增 `StorageWorkScheduler`
   - key：volume/library root，例如 `E:\`。
   - token：按 `storage_profile` 分配。
   - 支持任务优先级。
   - 支持任务取消、暂停、恢复。

2. 定义任务优先级
   - `reader`: 最高优先级。
   - `scan_fast`: 中优先级。
   - `metadata_scan`: 中低优先级。
   - `cover_build`: 低优先级。
   - `cache_write`: 低优先级，用于批量缩略图和派生缓存写入。
   - `identity_hash`: 最低优先级。

3. 接入重 IO 入口
   - 打开归档。
   - 获取页清单。
   - 提取封面。
   - 计算 quick/full hash。
   - 重建缩略图。
   - 缩略图写入缓存目录。

4. 阅读抢占
   - 有阅读请求进入时，暂停或延后同盘低优先级任务。
   - 后台任务释放 token 后阅读优先获得 token。
   - `idle_only_heavy_tasks` 开启时，后台封面、hash 等重 IO 只有在同盘没有活跃 token 时才启动，避免多个后台任务以不同并发上限绕开同盘保护。

### 验收标准

- 同一外接 HDD 同时最多 1 个后台重 IO 任务。
- 阅读请求等待后台任务的时间可记录。
- 后台任务支持取消和恢复。
- 扫描与封面重建同时触发时不会互相放大磁盘队列。
- 诊断接口能展示后台等待 token 数和当前暂停原因。

## 阶段 3：扫描低冲击分级

目标：日常增量扫描尽量只做文件系统比较，不打开归档、不计算全量 hash。

### 改造项

1. `fast_scan`
   - 遍历目录。
   - 比较 path、mtime、size。
   - 标记新增、删除、变化文件。
   - 未变化文件不打开归档。

2. `metadata_scan`
   - 仅对新增或变化文件打开归档。
   - 读取页数、ComicInfo、首图信息。
   - 产出缺封面任务，不同步生成封面。

3. `identity_scan`
   - quick hash 只对需要身份识别的文件执行。
   - full hash 默认后台低优先级。
   - KOReader 未启用时不计算 KOReader fingerprint。

4. `repair_scan`
   - 用户明确选择时才强制打开所有归档。
   - 明确提示会长时间占用磁盘。

### 验收标准

- 未变化文件扫描不会 `OpenArchive`。
- 外接 HDD 增量扫描不会触发封面同步重建。
- KOReader 关闭时不会计算 fingerprint/full hash。
- 扫描日志记录 `discovered_archives`、`skipped_archives`、`opened_archives`、`hashed_files`。

## 阶段 4：封面生成队列化与低速模式

目标：重建封面不再把机械盘随机 IO 打满。

### 改造项

1. 封面任务入队
   - 扫描只登记缺失或过期封面。
   - 后台 worker 按队列生成。
   - 支持任务暂停、取消、重试。
   - 全量缩略图重建作为可取消任务运行，任务完成点等待全库扫描和封面队列收尾。

2. 按存储设备限流
   - 外接 HDD 默认 `cover_concurrency = 1`。
   - 同一 volume 不并发打开多个归档。
   - SSD 可单独提高并发。

3. 阅读时暂停
   - 阅读同盘漫画时暂停封面生成。
   - 阅读停止一段时间后恢复。

4. 缩略图写入策略
   - 缩略图写入应用数据目录或 SSD cache。
   - 避免写回漫画所在外接机械盘。
   - 批量缩略图写入进入 `cache_write` token。
   - 缓存目录与漫画库同盘时复用同盘低冲击策略。
   - 记录缩略图写入耗时，便于定位大量小文件落盘造成的卡顿。

### 验收标准

- 重建封面期间手动浏览硬盘目录不会长期无响应。
- 阅读时封面 worker 自动降速或暂停。
- 封面任务可在任务中心取消。
- 封面完成后前端能增量刷新或下次进入时显示。

## 阶段 5：缓存策略与同盘写入控制

目标：避免在机械漫画盘上制造额外写入和元数据更新。

### 改造项

1. 页面磁盘缓存按库控制
   - 外接 HDD 策略默认禁用页面磁盘缓存。
   - 允许高级用户手动开启。

2. 缩略图与派生文件位置检查
   - 默认放应用数据目录。
   - 如果应用数据目录也在外接 HDD，给出诊断提醒。

3. 缓存写入限速
   - 对大批量缩略图写入增加批处理和间隔。
   - 避免频繁 fsync 或小文件风暴。
   - 记录缓存写入等待、暂停等待与实际写入耗时。

### 验收标准

- 外接 HDD 资源库默认不写页面磁盘缓存。
- 缩略图不会默认写到漫画原盘目录。
- 诊断页能提示 cache/data 是否与漫画库位于同一慢速盘。

## 阶段 6：后台任务用户体验

目标：用户能清楚知道后台任务对磁盘的影响，并能控制它。

### 改造项

1. 任务中心增强
   - 显示任务访问的资源库。
   - 显示存储策略。
   - 显示当前是否因阅读暂停。
   - 显示已打开归档数、已 hash 文件数、累计 IO 等待耗时、暂停耗时和预计剩余量。
   - 扫描完成后把 `opened_archives`、`hashed_files`、`io_wait_ms`、`paused_ms`、`thumbnail_write_ms` 写入任务状态，文件身份重建任务同步聚合 hash、IO 等待与暂停耗时。

2. 低冲击运行模式
   - 全局开关：“后台维护低冲击运行”。
   - 支持“仅空闲时运行”。
   - 支持“暂停全部后台 IO”。

3. 危险操作提示
   - 全量重建封面、repair scan、full hash 前提示会占用外接盘。
   - 提供“低速后台执行”和“立即执行”两个路径。

### 验收标准

- 用户能一键暂停扫描/封面/hash。
- 用户能看到任务是否正在等待 IO token。
- 全量重建类任务有明确的低冲击选项。

## 阶段 7：诊断与基准

目标：把“硬盘 100%”拆解成可定位的任务、资源库和 IO 类型。

### 改造项

1. 后端指标
   - 每个任务记录打开归档次数。
   - 每个任务记录读取文件数和 hash 文件数。
   - 记录队列等待时间、执行时间、暂停时间。
   - 记录阅读请求是否等待后台 IO。

2. 性能面板
   - 展示当前活跃 IO 任务。
   - 展示每个资源库的重 IO 并发。
   - 展示最近扫描/封面任务的归档打开速率。
   - 展示最近缩略图写入耗时，定位小文件写入压力。

3. 基准脚本
   - 外接 HDD 样本库扫描基线。
   - 封面重建基线。
   - 扫描与阅读并发基线。
   - SSD 与 HDD 策略对比。
   - `cmd/storageiobench` 采集目录遍历、抽样读取、并发读取和缓存小文件写入结果。
   - `cmd/storageiobench` 采集无调度后台读取与 reader-priority 低冲击调度下的阅读探针 P50/P95/Max。
   - 报告包含 `label/profile/notes`，支持同一格式记录 SSD 与外接 HDD 对比。
   - 报告根据吞吐、阅读延迟和小文件写入自动生成并发与缓存位置建议。
   - 基线结果保存到 `docs/performance-baselines/`。

### 验收标准

- 能从诊断页判断是扫描、封面、hash 还是阅读在访问磁盘。
- 能证明低冲击模式下阅读请求延迟下降。
- 能记录外接 HDD 上扫描/封面任务的基线数据。

## 推荐交付顺序

| 顺序 | 阶段 | 原因 |
| --- | --- | --- |
| 1 | 阶段 1 存储介质策略与配置模型 | 先让用户能声明外接机械盘场景 |
| 2 | 阶段 2 统一存储 IO 调度器 | 解决扫描、封面、阅读互相抢盘 |
| 3 | 阶段 3 扫描低冲击分级 | 降低日常增量扫描 IO |
| 4 | 阶段 4 封面生成队列化与低速模式 | 直接缓解重建封面卡盘 |
| 5 | 阶段 5 缓存策略与同盘写入控制 | 避免额外写入慢盘 |
| 6 | 阶段 6 后台任务用户体验 | 让用户可见、可停、可控 |
| 7 | 阶段 7 诊断与基准 | 固化结果，避免回退 |

## 第一批建议落地范围

第一批优先实现低风险、高收益的控制面：

1. 为资源库增加 `storage_profile` 与 `io_policy` 配置。
2. 外接 HDD 默认将扫描归档打开、封面生成、hash 并发限制为 1。
3. 封面重建从同步批处理改为可暂停队列。
4. 阅读时暂停同盘封面/hash 任务。
5. 页面磁盘缓存对外接 HDD 默认关闭。
6. 任务日志增加 `volume_key`、`storage_profile`、`opened_archives`、`hashed_files`、`io_wait_ms`。
7. 任务日志增加 `paused_ms` 与 `thumbnail_write_ms`，用于区分主动让出磁盘和真实写入耗时。
8. 缩略图重建改为可取消低冲击任务，任务中心显示真实运行状态。
9. 增加 `cmd/storageiobench`，输出外接盘 IO 基线 Markdown。

这批改造可以先不改变归档解析和图片处理核心逻辑，但能显著降低机械盘被后台任务打满时的交互卡顿。
